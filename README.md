## Математическая постановка задачи

### 1. Модель BoT-задачи

Рассматривается вычислительная задача типа **bag-of-tasks**: общий объём работы (W) (в вашем случае — общее число Монте-Карло семплов) может быть разбит на независимые подзадачи (чанки). Каждый чанк выполняет (g_k) семплов и возвращает частичный результат (например, сумму значений функции или число попаданий), после чего результаты агрегируются.

Пусть требуется оценить (d)-мерный интеграл
[
I=\int_{[a,b]^d} f(\mathbf{x}), d\mathbf{x}.
]
Метод Монте-Карло строит оценку
[
\hat I(W)=V\cdot\frac{1}{W}\sum_{n=1}^{W} f(\mathbf{x}_n),\quad V=(b-a)^d,
]
где (\mathbf{x}_n) равномерно распределены на ([a,b]^d).

При разбиении на чанки (k=1..K) (где (\sum_{k=1}^{K} g_k=W)) каждый чанк возвращает сумму
[
S_k=\sum_{n=1}^{g_k} f(\mathbf{x}*{k,n}).
]
Тогда агрегированная оценка:
[
\hat I = V\cdot\frac{\sum*{k=1}^{K} S_k}{\sum_{k=1}^{K} g_k}=V\cdot\frac{\sum_{k=1}^{K} S_k}{W}.
]
Это важный момент для BoT: **точность определяется только (W)**, а планировщик влияет на **время выполнения**, т.е. на makespan.

---

### 2. Модель гетерогенного кластера и времени выполнения

Пусть Kubernetes-кластер состоит из узлов (n\in\mathcal{N}) с различной вычислительной мощностью и фоновыми нагрузками. На узлах запущены воркеры (pod’ы), каждый выполняет чанки последовательно.

Обозначим:

* (K) — число чанков (или число pod’ов в статическом режиме),
* (g_k) — размер (k)-го чанка,
* (t_k) — время выполнения (k)-го чанка (измеряется в `duration_ms`),
* (T) — время завершения всего прогона (**makespan**, `WALL_MS`),
* (T_{seq}) — время последовательного запуска.

Цель планирования: **минимизировать makespan**
[
\min T,
]
при ограничениях:
[
\sum_{k=1}^K g_k = W,\qquad g_k\in\mathbb{Z}_{>0}.
]

---

### 3. Метрики сравнения

Для каждого эксперимента фиксируется:

1. **Makespan**: (T) (wall time)
2. **Speedup** относительно последовательного:
   [
   S = \frac{T_{seq}}{T}.
   ]
3. (опционально) **Эффективность** относительно количества CPU/воркеров (P):
   [
   E=\frac{S}{P}.
   ]
4. **Straggler metric** (хвост):
   [
   \text{StragglerRatio}=\frac{\max_k t_k}{\operatorname{mean}_k t_k},
   ]
   или просто `Max chunk duration` (у тебя уже есть).
5. **Справедливость распределения по узлам** (для гетерогенности): доля семплов, выполненных узлом (n),
   [
   p_n=\frac{W_n}{W}.
   ]
   В идеале (p_n) близко к доле пропускной способности узла (оцененной экспериментально или по числу CPU).

# Эксперимент 3. Статическое распределение по узлам: равномерное и пропорциональное CPU

**Цель:** показать, что в гетерогенном кластере “равное” распределение ведёт к хвосту на слабых узлах, а пропорциональное — лучше.

**Сценарии:**

* Total (K=40) (или другой фиксированный), но чанки распределяются *по узлам заранее*.
* Вариант A (“supernaive”): 13/13/14
* Вариант B (“cpu proportional”): 6/11/23 (пропорционально 2/4/8)

**Что сравнивать:**

* makespan (T): он будет определяться самым медленным узлом (тот, кому дали “слишком много”).
* длительность job на каждом узле.

**Как красиво описать математически:**

* При статическом разбиении makespan приближенно:
  [
  T \approx \max_{n\in\mathcal{N}}\frac{W_n}{V_n},
  ]
  где (W_n) — работа, назначенная узлу (n), (V_n) — его пропускная способность.
  Равномерное распределение (W_n\approx W/|\mathcal{N}|) плохо при разных (V_n). Пропорциональное (W_n\propto V_n) уменьшает максимум.
