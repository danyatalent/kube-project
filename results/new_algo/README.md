## Модель задачи Bag-of-Tasks

Пусть требуется вычислить значение функционала (например, многомерного интеграла методом Монте-Карло) при фиксированном общем числе выборок (W) (samples).
Вычисление представляется как Bag-of-Tasks: задача разбивается на независимые подзадачи (чанки), каждая из которых выполняет (g) семплов и возвращает частичную статистику.

Есть набор воркеров ( \mathcal{I}={1,\dots,m}). Воркеры работают в pull-модели: после завершения чанка воркер запрашивает следующий.

Целевая метрика: **makespan** (время выполнения всего прогона)

[
T = \max_{i\in \mathcal{I}} T_i,
]

где (T_i) — суммарное время, затраченное воркером (i) на выполненные им чанки в рамках прогона (с учётом конкуренции и фоновой нагрузки узлов).

---

## Алгоритм исполнения: централизованный диспетчер и pull-воркеры

Диспетчер хранит состояние прогона:

* общий объём работы (W),
* остаток работы (R) (в коде `remainW`),
* объём уже выданной работы (A) (`assigned`),
* объём подтверждённо выполненной работы (D) (`doneW`).

В момент старта прогона:
[
R \leftarrow W,\quad A \leftarrow 0,\quad D \leftarrow 0.
]

Каждый воркер циклически выполняет:

1. `GET /task` → получить чанк ((task_id, g, seed)) или сигнал “работа закончена”
2. вычислить чанк (one-shot бинарник), измерить длительность (t)
3. `POST /result` → отправить результат + измерения ((g,t))

---

## Базовый режим Fixed (динамическое self-scheduling)

В режиме `fixed` каждому запросу воркера выдаётся фиксированный размер чанка (g=g_0) (параметр `ChunkSamples`), ограниченный остатком:

[
g \leftarrow \min(g_0, R).
]

После выдачи:
[
R \leftarrow R - g,\quad A \leftarrow A + g.
]

Это **динамическое балансирование** за счёт pull-модели: быстрые воркеры запрашивают задачи чаще и выполняют больше чанков.

## Режим Adaptive: online-оценка скорости и адаптивный выбор чанка

### 1) Online-оценка производительности воркера

После выполнения чанка воркер (i) возвращает:

* (g) — число семплов в чанке
* (t) — измеренная длительность чанка (в секундах; в коде `DurationMS`)

Мгновенная оценка скорости:

[
\tilde v_i = \frac{g}{t} \quad [\text{samples/sec}].
]

Диспетчер поддерживает сглаженную оценку скорости (EMA):

[
\hat v_i \leftarrow (1-\alpha)\hat v_i + \alpha \tilde v_i,
]

где (\alpha\in(0,1]) — коэффициент сглаживания (`Alpha`).
При старте (до первых измерений) используется априорное значение (\hat v_i \gets v_{prior}) и для первого чанка выдаётся минимальный размер (warm-up).

> В коде: `speedHat[workerID]`, `tilde := samples / time`, EMA с `Alpha`, и warm-up через `MinChunk`.

Дополнительно применяется ограничение выбросов (clamp) для (\tilde v_i):

[
\tilde v_i \leftarrow \min(\max(\tilde v_i, \hat v_i/3), 3\hat v_i),
]

чтобы единичные аномальные измерения не разрушали оценку.

---

### 2) Выбор размера следующего чанка (взвешенное распределение остатка)

Пусть текущий остаток работы (R), а множество воркеров, для которых есть оценки, имеет суммарную оценочную производительность:

[
S = \sum_{j \in \mathcal{I}} \hat v_j.
]

Тогда воркеру (i) при запросе выдаётся размер чанка:

[
g_i^* = \beta \cdot R \cdot \frac{\hat v_i}{S},
]

где (\beta\in(0,1]) — параметр “агрессивности” (`Beta`).
Далее применяется дискретизация и ограничения:

[
g_i = \mathrm{clip}\Big(\mathrm{round}(g_i^*),; g_{min},; g_{max}^{eff}\Big),
]
[
g_i \leftarrow \min(g_i, R),
]

где:

* (g_{min}=) `MinChunk`,
* (g_{max}^{eff}=\min(g_{max}, g_{tail}(R))) — эффективный максимум с учётом tail-ограничения (см. ниже),
* `clip(x, a, b)` ограничивает (x) интервалом ([a,b]).

После выдачи:
[
R \leftarrow R - g_i,\quad A \leftarrow A + g_i.
]

Интерпретация: остаток работы (R) распределяется пропорционально измеренной скорости (\hat v_i), что приближает выполнение к “идеальному” случаю (g_i \propto v_i) при неизвестной гетерогенности.

---

### 3) Tail-aware ограничение максимального чанка (уменьшение хвоста)

Для снижения эффекта “длинного хвоста” (straggler) при малом остатке вводится автоматическое ограничение максимального чанка:

Пусть доля оставшейся работы:

[
\rho = \frac{R}{W}.
]

Тогда:

[
g_{tail}(R) =
\begin{cases}
g_{max}/4, & \rho \le 0.02,\
g_{max}/2, & \rho \le 0.10,\
g_{max}, & \text{иначе}.
\end{cases}
]

И используется (g_{max}^{eff} = \min(g_{max}, g_{tail}(R))).

Смысл: ближе к завершению прогон дробится на более мелкие чанки, чтобы уменьшить вероятность того, что один крупный чанк на “шумном” узле определит makespan.

## Спекулятивное выполнение (speculative execution) на хвосте

После того как весь объём работы выдан ((R\le 0)), но результаты ещё не собраны ((D<W)), диспетчер может повторно выдать один из “подозрительно долгих” незавершённых чанков второму воркеру.

Для оценки “долго” диспетчер ведёт статистику длительностей чанков (online mean/std). Порог:

[
\theta = \max(20000,; \mu + 2\sigma)\quad [\text{ms}],
]

где (\mu,\sigma) — среднее и стандартное отклонение длительностей ранее завершённых чанков.

Далее выбирается чанк, который:

* ещё не завершён,
* был выдан ровно один раз,
* “висит” дольше порога (\theta).

Его повторно выдают (тот же `task_id`, тот же `seed`, тот же (g)).
При получении результатов:

* первый пришедший результат засчитывается,
* последующие дубликаты игнорируются (по `task_id`).

Эта техника уменьшает вклад редких stragglers в makespan при цене небольшого перерасхода вычислений на хвосте.

---

## Критерий завершения прогона

Прогон считается завершённым, когда весь объём работы выдан и подтверждённо выполнен:

[
A \ge W ;;\wedge;; D \ge W.
]

---

## Что можно подчеркнуть в отчёте

1. **Само-балансировка (pull)**: динамическое распределение делает систему устойчивой к гетерогенности и шуму.
2. **Online performance estimation**: (\hat v_i) измеряется, а не задаётся по CPU.
3. **Weighted adaptive chunking**: выдача (g_i) пропорционально (\hat v_i) стремится к выравниванию времени занятости воркеров.
4. **Tail reduction**: ограничение (g_{max}) по доле остатка + speculative execution снижают эффект “длинного хвоста”.
